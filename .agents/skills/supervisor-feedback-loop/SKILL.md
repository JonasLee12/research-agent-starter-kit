# Supervisor Feedback Loop

Use when the user provides supervisor comments, examiner comments, peer review, or asks to revise a chapter based on feedback.

## Workflow

1. Separate each feedback item into:
   - required change
   - optional suggestion
   - clarification question
   - disagreement or tradeoff
2. Prioritize by dissertation risk:
   - research design / validity
   - ethics / privacy
   - evidence support
   - structure / coherence
   - style / formatting
3. Map each item to files and sections.
4. Propose a revision plan before large edits.
5. For major proposal/chapter revisions, create a dissertation revision matrix before editing:
   - current function,
   - needed function,
   - motivation link,
   - operation: `KEEP`, `REWRITE`, `SPLIT`, `MERGE`, `MOVE`, `DELETE`, or `ADD`,
   - evidence source,
   - remaining supervisor decision.
6. If asked to edit, keep changes scoped and preserve the user's voice.
7. After a major rewrite, run a logic-transfer audit before calling the revision complete.
8. Update `research-wiki/SUPERVISOR_FEEDBACK.md` or `audit-reports/FEEDBACK_RESPONSE.md`.

## Output

For each feedback item, record:

- feedback summary
- action taken or recommended
- file/section affected
- status: open, addressed, deferred, or question

For major revisions, also record:

- revision matrix location,
- logic-transfer audit verdict,
- unresolved supervisor questions.

Use `../dissertation-argument-spine/references/paperspine-adapted-writing-controls.md` for the revision matrix and logic-transfer audit templates when needed.

## Guardrails

Do not silently ignore critical feedback. If feedback conflicts, surface the conflict and recommend how to ask the supervisor.
